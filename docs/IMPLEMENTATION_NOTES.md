# Implementation Notes

Цей документ пояснює основні рішення в MVP для охочуючих, котрі тільки знайомиться з кодовою базою.

## Навіщо RTK Query, якщо дані локальні?

UI використовує hooks на кшталт `useGetTasksQuery` і `useCreateTaskMutation`.
Зараз ці hooks читають і записують `localStorage` через fake API adapter.

Це дає стабільну межу для роботи з даними:

- компоненти не знають про `localStorage`;
- локальний adapter пізніше можна замінити на `fetchBaseQuery`;
- більшість UI-коду може залишитись без змін, коли з'явиться реальний backend.

Це frontend-friendly варіант підходу Data Access Layer / Adapter.

## Чому статус задачі не зберігається

Об'єкт задачі зберігає факти:

- дату;
- час початку;
- час завершення;
- назву.

Статус обчислюється з цих фактів і поточного часу.
Це запобігає багам, коли збережений статус стає застарілим.

## Чому є один time ticker

Якби кожна task card мала власний `setInterval`, 20 задач означали б 20 таймерів.
Замість цього застосунок має один `CurrentTimeProvider`.
Task cards читають одне й те саме значення часу та на його основі обчислюють свій display state.

## Чому date/time логіка винесена в helpers

Коли правила часу зберігаються в pure functions, їх:

- легше читати;
- легше тестувати;
- можна перевикористовувати між form validation і UI rendering.

## Межі MVP

Застосунок свідомо не включає drag-and-drop, resizing задач, week strip або реальний backend.
Це хороші post-release features, але вони відволікали б від основної суті тестового завдання.
